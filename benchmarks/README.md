# benchmarks/

Reproducible microbenchmark harness for the Trinary engine kernels.

## Files

| File              | Purpose                                                   |
|-------------------|-----------------------------------------------------------|
| `bench_engine.c`  | C harness: warmup + N replicates per kernel, JSON output. |
| `perf_gate.py`    | Diff tool: fails CI if any kernel regresses > tolerance.  |
| `baseline.json`   | Author-machine reference run (regenerated by maintainer). |

## How it works

1. `build.ps1` / `build.sh` compiles `bench_engine.c` into
   `build/bin/trinary-bench(.exe)` statically linked against `libtrinary.a`.
2. The harness does `WARMUP` untimed runs, then `REPLICATES` timed runs, then
   reports per-kernel `min / median / mean / max / stddev` seconds and
   best-case throughput in a strictly versioned JSON schema
   (`"schema": "trinary.bench.v1"`).
3. `perf_gate.py` loads a current JSON and a baseline JSON, and exits non-zero
   if any named kernel's best-case throughput is worse than
   `baseline * (1 - tolerance)`. Default tolerance = 3 % (decision D-007).
4. Current gated kernels in `baseline.json`: `braincore_u8`, `harding_gate_i16`,
   `lattice_flip`, and `prime_sieve`.

## Running locally

```powershell
./build.ps1 -Config Release
./build/bin/trinary-bench.exe > build/bench/latest.json
python benchmarks/perf_gate.py --baseline benchmarks/baseline.json --current build/bench/latest.json
```

## Refreshing the baseline

Only the repository maintainer may refresh `baseline.json`, and only on the
designated reference machine, after running on a quiesced system. The
refresh is a decision event (log it in `trinary-tripy-architecture.md §8`).
